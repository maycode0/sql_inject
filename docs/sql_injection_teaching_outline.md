# SQL 注入教师授课版大纲

## 1. 课程定位

### 1.1 课程主题

SQL 注入基础原理、风险识别与安全防御。

### 1.2 适用对象

- 计算机相关专业学生
- Web 开发初学者
- 后端开发工程师
- 安全基础学习者

### 1.3 授课目标

完成本课程后，学员应能够：

1. 理解 SQL 注入的定义、本质与危害。
2. 识别常见的脆弱 SQL 拼接代码。
3. 区分值位置和结构位置的风险。
4. 掌握参数化查询与白名单映射两类核心防御方法。
5. 能基于本地演示程序完成基础观察与分析。

---

## 4. 授课大纲

### 第一部分：课程导入

授课目标：

- 激发学习兴趣
- 建立“安全编码”视角

讲解要点：

- 为什么数据库安全重要
- 为什么 SQL 注入是经典又常见的安全问题
- 本课程重点是“理解与防御”，不是攻击技巧堆砌

教师可直接使用的导入语：

“今天我们要学习的不是如何攻击系统，而是为什么一些常见代码写法会带来 SQL 注入风险，以及开发者应如何从代码层面把风险消除掉。”

---

### 第二部分：SQL 注入基础概念

授课目标：

- 让学员理解 SQL 注入是什么
- 明白 SQL 注入的本质原因

讲解要点：

- SQL 注入定义
- 本质原因：把用户输入当成 SQL 代码结构的一部分
- 常见危害：绕过验证、读取数据、修改数据、影响业务

板书关键词建议：

- 输入
- 拼接
- SQL 结构
- 参数化

教师提问示例：

1. SQL 注入的根本问题是什么？
2. 风险来自输入本身，还是来自拼接方式？

---

### 第三部分：脆弱代码识别

授课目标：

- 让学员能识别最基础的危险代码

讲解要点：

- 字符串拼接 SQL 的典型形式
- 为什么登录、搜索、ID 查询都可能有风险
- 为什么 `LIKE` 查询仍然可能有风险

示例讲解代码：

```python
query = "SELECT * FROM users WHERE username = '" + username + "'"
```

```python
query = f"SELECT * FROM users WHERE id = {user_id}"
```

教师强调点：

- 不要只盯着单引号
- 不要把风险理解成某个特殊字符
- 真正问题是“数据和代码没有分离”

---

### 第四部分：参数化查询

授课目标：

- 让学员掌握值位置的核心防御方式

讲解要点：

- 什么是参数化查询
- 为什么参数化更安全
- 值位置的典型场景：用户名、密码、ID、关键字

示例代码：

```python
query = "SELECT * FROM users WHERE username = ? AND password = ?"
cursor.execute(query, (username, password))
```

教师强调点：

- 参数化的核心是让输入作为数据处理
- 参数化并不意味着所有动态 SQL 场景都能直接解决

---

### 第五部分：白名单映射

授课目标：

- 让学员理解结构位置为什么需要白名单

讲解要点：

- 什么是结构位置
- 常见结构位置：表名、列名、排序字段、排序方向
- 为什么 `ORDER BY` 常常要做白名单映射

示例代码：

```python
allowed_fields = {
    "id": "id",
    "username": "username",
}

safe_field = allowed_fields.get(order_by, "id")
query = f"SELECT id, username FROM users ORDER BY {safe_field}"
```

教师强调点：

- 值位置和结构位置要分开讲
- 结构位置最重要的思想是“只允许有限、明确、可控的片段进入 SQL”

---

### 第六部分：课堂程序演示

授课目标：

- 通过实际运行帮助学员建立直观理解

使用文件：

- `docs/sql_injection_demo.py`

演示步骤：

1. 运行登录示例
2. 展示脆弱写法输出的 SQL
3. 展示安全写法输出的 SQL 模板和参数
4. 运行排序示例
5. 展示白名单如何限制排序字段

运行命令：

```bash
python docs/sql_injection_demo.py
```

或：

```bash
py docs/sql_injection_demo.py
```

教师讲解重点：

- 登录示例讲“值位置”
- 排序示例讲“结构位置”
- 两个示例结合起来，帮助学员建立完整防御思路

---

### 第七部分：防御清单总结

授课目标：

- 帮助学员形成工程化防御意识

讲解要点：

- 所有查询默认参数化
- 排序字段等动态结构默认白名单
- 输入做类型和长度校验
- 数据库权限坚持最小权限
- 错误信息统一处理，不直接暴露底层报错
- 日志记录关键失败行为

教师总结语建议：

“防 SQL 注入不是记几个特殊情况，而是建立一个稳定的开发习惯：值位置参数化，结构位置白名单，权限最小化，错误统一处理。”

---

### 第八部分：课堂练习与讲评

授课目标：

- 通过练习检查理解效果

使用文件：

- `docs/sql_injection_exercises.md`

建议课堂练习题：

1. SQL 注入的根本原因是什么？
2. 为什么字符串拼接 SQL 有风险？
3. 为什么排序字段通常要做白名单？
4. 请找出给出的代码中风险点。

讲评方式建议：

- 先让学员独立回答
- 再让学员解释“为什么”
- 最后教师统一总结正确思路

## 5. 课堂互动设计

### 互动问题 1

“如果一段代码把用户输入直接拼进 SQL，即使没有明显单引号，也一定安全吗？”

预期引导：

- 让学员意识到风险不只出现在单引号场景

### 互动问题 2

“排序字段为什么不能简单理解成普通输入值？”

预期引导：

- 帮助学员理解结构位置的概念

### 互动问题 3

“ORM 是否等于绝对安全？”

预期引导：

- 帮助学员理解 ORM 只是降低风险，不等于消灭风险

## 7. 评价标准建议

### 优秀

- 能准确解释 SQL 注入本质
- 能独立区分值位置和结构位置
- 能给出正确防御方案

### 合格

- 能理解 SQL 注入原理
- 能识别常见拼接风险
- 能理解参数化查询的重要性

### 需加强

- 只记住表面案例，不能解释本质
- 不能区分参数化与白名单的适用场景
- 仍把风险简单理解为“某个字符危险”

---

## 8. 授课结束总结

教师可使用如下结束语：

“今天最重要的不是记住某段输入，而是记住一个原则：只要用户输入进入 SQL 结构，就要警惕风险。值位置优先参数化，结构位置优先白名单，数据库权限坚持最小权限，这就是我们写安全数据库代码的基本方法。”
