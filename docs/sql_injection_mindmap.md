# SQL 注入知识脑图版总结

## 1. 总览脑图

```text
SQL 注入
├─ 1. 定义
│  ├─ 用户输入进入 SQL 语句
│  ├─ 输入不再只是数据
│  └─ 可能改变 SQL 原始逻辑
├─ 2. 本质原因
│  ├─ 字符串拼接 SQL
│  ├─ 没有使用参数化查询
│  └─ 动态结构缺少白名单控制
├─ 3. 常见危害
│  ├─ 绕过登录
│  ├─ 读取敏感数据
│  ├─ 修改或删除数据
│  ├─ 影响业务可用性
│  └─ 扩大攻击面
├─ 4. 常见入口
│  ├─ 登录表单
│  ├─ 搜索框
│  ├─ URL 参数
│  ├─ API 参数
│  ├─ 分页参数
│  ├─ 排序字段
│  └─ 报表筛选条件
├─ 5. 风险位置分类
│  ├─ 值位置
│  │  ├─ username
│  │  ├─ password
│  │  ├─ id
│  │  └─ keyword
│  └─ 结构位置
│     ├─ 表名
│     ├─ 列名
│     ├─ 排序字段
│     └─ 排序方向
├─ 6. 核心防御
│  ├─ 参数化查询
│  ├─ 白名单映射
│  ├─ 输入校验
│  ├─ 最小权限
│  ├─ 异常控制
│  └─ 日志审计
├─ 7. 不可靠做法
│  ├─ 只过滤关键字
│  ├─ 只过滤单引号
│  ├─ 只做前端校验
│  └─ 认为 ORM 一定安全
├─ 8. 学习路径
│  ├─ 初级：理解原理
│  ├─ 中级：识别风险并修复
│  └─ 高级：代码审计和体系化防护
└─ 9. 关键结论
   ├─ 本质是输入变成了 SQL 代码
   ├─ 值位置优先参数化
   ├─ 结构位置优先白名单
   └─ 权限控制坚持最小权限
```

## 2. 核心概念脑图

```text
核心概念
├─ SQL 注入是什么
│  ├─ 输入被拼接进 SQL
│  ├─ SQL 含义被改变
│  └─ 查询结果偏离预期
├─ 为什么会发生
│  ├─ 开发者图省事直接拼接
│  ├─ 缺少参数绑定
│  └─ 动态字段控制不严格
└─ 怎么看待它
   ├─ 不是某个字符本身危险
   ├─ 而是拼接方式危险
   └─ 核心是数据和代码没有分离
```

---

## 3. 风险形成过程脑图

```text
风险形成过程
├─ 用户提交输入
├─ 后端接收输入
├─ 程序拼接 SQL
├─ 输入进入 SQL 文本
├─ 数据库按完整 SQL 解释
└─ 导致结果偏离预期
```

可以记成一句话：

- 输入进入 SQL 文本并参与解释，风险就出现了。

## 5. 值位置与结构位置脑图

```text
输入位置分类
├─ 值位置
│  ├─ WHERE username = ?
│  ├─ WHERE id = ?
│  ├─ LIKE ?
│  └─ INSERT INTO ... VALUES (?, ?)
│
│  处理方式
│  └─ 参数化查询
│
└─ 结构位置
   ├─ ORDER BY username
   ├─ SELECT column_name
   ├─ FROM table_name
   └─ ASC / DESC

   处理方式
   └─ 白名单映射
```

结论：

- 值位置主要防拼接
- 结构位置主要防越权进入 SQL 结构
