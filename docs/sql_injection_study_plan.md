# SQL 注入初级 / 中级 / 高级三阶段学习计划

## 1. 学习计划概览

这份计划适合按阶段逐步推进，目标不是记忆零散技巧，而是建立一套完整认知：

- 初级阶段：理解 SQL 注入是什么、为什么会发生。
- 中级阶段：能够识别常见风险点，并写出基础防御代码。
- 高级阶段：具备代码审计、复杂场景防护和安全验证思维。

建议学习周期：

- 初级：3 到 5 天
- 中级：5 到 7 天
- 高级：7 到 14 天

可根据你的基础灵活调整。

---

## 2. 初级阶段：建立基础认知

### 2.1 学习目标

完成初级阶段后，你应该能够：

1. 说清 SQL 注入的定义。
2. 理解 SQL 是如何与用户输入混在一起产生风险的。
3. 看懂最基础的字符串拼接漏洞代码。
4. 理解参数化查询为什么更安全。

### 2.2 必学知识点

- SQL 基础语法
  - `SELECT`
  - `WHERE`
  - `INSERT`
  - `UPDATE`
  - `DELETE`
- Web 应用访问数据库的基本流程
- 用户输入进入后端、进入数据库查询的路径
- 字符串拼接构造 SQL 的风险
- 参数化查询 / 预编译语句的基本概念

### 2.3 推荐学习内容

- 阅读 `docs/sql_injection_learning.md`
- 手写几条最基础的 SQL 查询
- 对比下面两种写法的区别：
  - 直接拼接 SQL
  - 参数化 SQL

### 2.4 实践任务

完成以下练习：

1. 自己写一个最简单的 `users` 表。
2. 写一个普通登录查询。
3. 再写一个参数化版本的登录查询。
4. 观察两种写法在结构上的差异。

### 2.5 阶段验收标准

达到以下标准即可进入中级阶段：

- 能解释 SQL 注入的根本原因。
- 能指出拼接 SQL 的危险点。
- 能独立写出一个参数化查询示例。

---

## 3. 中级阶段：识别风险与完成基础防护

### 3.1 学习目标

完成中级阶段后，你应该能够：

1. 在真实业务代码里识别常见注入风险点。
2. 理解不只是登录框，搜索、分页、排序也可能有风险。
3. 知道哪些位置适合参数化，哪些位置必须白名单。
4. 能设计一套基础防御方案。

### 3.2 必学知识点

- 常见输入入口
  - 表单输入
  - URL 参数
  - API 请求体
  - 报表筛选条件
- 常见风险场景
  - 登录验证
  - 搜索查询
  - ID 查询
  - 动态排序
  - 动态列名或表名
- 黑名单过滤为什么不可靠
- 输入校验与类型约束
- 数据库最小权限原则
- 错误信息隐藏与日志审计

### 3.3 推荐重点理解

这个阶段最关键的是建立下面两个判断：

- 用户输入作为“值”时，优先参数化。
- 用户输入影响“结构”时，如排序字段、列名、表名，要做白名单映射。

### 3.4 实践任务

建议完成以下练习：

1. 写一个搜索功能，先用拼接方式实现，再改成参数化方式。
2. 写一个带排序参数的列表功能。
3. 对排序字段增加白名单，例如仅允许：`name`、`created_at`。
4. 对分页参数做整数校验。
5. 模拟数据库报错，观察应该如何统一处理异常。

### 3.5 阶段验收标准

达到以下标准即可进入高级阶段：

- 能从业务代码中指出多个潜在注入入口。
- 知道参数化不是万能的，白名单同样重要。
- 能写出一版更完整的基础防护代码。

---

## 4. 高级阶段：代码审计、复杂场景与系统化安全思维

### 4.1 学习目标

完成高级阶段后，你应该能够：

1. 在较大项目中快速定位高风险 SQL 构造点。
2. 理解 ORM、查询构造器、原生 SQL 混用时的风险。
3. 识别复杂动态查询中的结构性风险。
4. 形成“开发 + 测试 + 审计”一体化安全思维。

### 4.2 必学知识点

- 代码审计中的危险模式识别
  - 字符串拼接 SQL
  - 模板字符串拼接查询条件
  - 动态拼接 `ORDER BY`
  - 动态拼接列名 / 表名
  - ORM 中直接执行原生 SQL
- 多层架构中的输入传递链
- 安全测试环境中的验证思路
- 数据权限与业务权限的边界
- 安全日志、告警与审计思路

### 4.3 推荐进阶方向

- 研究不同语言数据库驱动如何实现参数绑定
- 学习常见 ORM 的安全边界
- 学习如何对旧系统进行风险排查和逐步整改
- 在合法靶场或实验环境中做修复验证

### 4.4 实践任务

建议完成以下练习：

1. 选一个小型项目，搜索所有 SQL 构造位置。
2. 标记其中哪些是值注入风险，哪些是结构注入风险。
3. 为每一个风险点给出修复方案。
4. 统一整理为一份“SQL 安全检查清单”。
5. 设计一个上线前检查流程，确保数据库访问代码经过审查。

### 4.5 阶段验收标准

达到以下标准说明你已具备较好的进阶能力：

- 能系统分析一个模块的 SQL 注入风险。
- 能区分“理论上危险”和“实际可利用”的差异。
- 能提出兼顾可维护性和安全性的修复建议。

## 5. 三阶段配套练习安排

### 第 1 周：初级阶段

- 学 SQL 基础查询
- 阅读 `docs/sql_injection_learning.md`
- 理解字符串拼接的风险
- 跑通一个本地参数化查询示例

### 第 2 周：中级阶段

- 给搜索、分页、排序场景做风险分析
- 写输入校验逻辑
- 写字段白名单映射
- 整理自己的防御笔记

### 第 3 周及以后：高级阶段

- 对一个小项目做静态审查
- 梳理数据库访问层的封装方式
- 检查是否存在原生 SQL 拼接
- 建立自己的安全审查清单

---

## 6. 推荐学习方法

### 理论学习

- 先理解原理，再背防御方法。
- 学每个知识点时，都问自己：这个输入最终是值，还是会影响 SQL 结构？

### 代码练习

- 每学一个概念，都写一个最小示例。
- 每个示例都同时保留“脆弱版”和“安全版”对照。

### 复盘总结

- 学完一个阶段，写一页自己的总结。
- 不要只记攻击形式，要总结修复思路。

为了真正学会，建议每个阶段都产出可复用内容：

### 初级阶段输出

- 一份 SQL 基础语法笔记
- 一份“SQL 注入是什么”的个人总结
- 一个参数化查询最小示例

### 中级阶段输出

- 一份常见风险点清单
- 一份排序字段白名单示例
- 一份基础防御规范草稿

### 高级阶段输出

- 一份代码审计记录
- 一份安全整改建议文档
- 一份数据库访问安全检查清单

---

## 8. 结论

如果你按这三阶段推进，学习效果会比零散看资料更稳定：

- 初级阶段解决“看懂原理”。
- 中级阶段解决“看懂代码风险并会修”。
- 高级阶段解决“能系统审查和设计防护方案”。

真正的目标不是记住几个案例，而是形成一种习惯：

- 只要看到 SQL 拼接，就主动怀疑风险。
- 只要遇到动态结构，就优先考虑白名单。
- 只要涉及数据库权限，就坚持最小权限原则。
