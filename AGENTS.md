# AGENTS.md

## 目的与适用范围

- 本文件面向在本仓库中工作的 agentic coding agents，目标是帮助代理快速理解项目结构、执行命令、遵循既有风格，并在修改时避免破坏教学资料的准确性与安全边界。
- 当前仓库以 Markdown 教学资料和一个 Python 演示脚本为主，不是完整的应用服务，也没有成熟的构建流水线。
- 因此，本文件优先记录“仓库当前真实存在的做法”，同时对缺失的工具链给出保守约定，避免代理臆造不存在的命令。
- 本仓库内容聚焦 SQL 注入教学、PPT/讲义资料整理，以及与演示流程相关的提示词和工作流文档。

## 仓库概览

- `ai_coding/sql_injection_demo.py`：唯一可直接运行的 Python 演示代码，展示脆弱 SQL 拼接、参数化查询、排序字段白名单。
- `ai_coding/README.md`：代码示例说明与运行方式。
- `docs/`：SQL 注入学习手册、演示说明、练习与教学大纲等资料。
- `references/`：提示词、工作方法、agent 集成说明。
- `.trae/skills/ppt-agent-workflow/SKILL.md`：Trae skill 规则，定义 PPT 代理的阶段化工作方法。
- `opencode.json`：OpenCode 配置文件，仅供本地工具使用，不应在改动中暴露或复制敏感信息。

## 构建、Lint、测试命令

### 当前可确认命令

- 运行主示例：`python ai_coding/sql_injection_demo.py`
- Windows Python Launcher 备用写法：`py ai_coding/sql_injection_demo.py`
- 从仓库根目录执行即可，无需额外参数。

### 当前仓库状态说明

- 没有正式的 build 命令；大多数内容是文档，不需要编译。
- 没有正式的 lint 命令；不要在说明中声称存在 Black、Ruff、Prettier 等流程，除非你先引入并配置。
- 没有正式的 test suite；当前只能通过运行演示脚本做手动验证。

### 单个测试的运行方式

- 当前仓库没有单元测试框架，因此不存在真正的“单个测试用例命令”。
- 如果只是验证 Python 示例中的某一类行为，推荐直接运行整个脚本并观察对应输出段落：`python ai_coding/sql_injection_demo.py`
- 如需临时验证某个函数，可在本地交互环境中导入函数后手工调用，但这属于临时开发手段，不应写成项目标准命令。
- 若后续引入 `pytest`，请在新增测试后同步更新本文件，明确记录类似 `pytest path/to/test_file.py::test_name` 的单测命令。

### 代理执行命令时的建议

- 优先从仓库根目录执行命令，避免依赖 IDE 工作目录。
- 在没有测试框架时，把“运行示例脚本 + 人工检查输出”视为最小验证步骤。
- 若你新增了脚本或测试，请同时补充 README 和本文件，不要让文档落后于代码。

## 代码风格总原则

- 保持教学导向：代码首先要便于讲解，其次才是技巧性。
- 保持最小复杂度：优先使用直接、可读、低魔法的实现。
- 保持本地可运行：示例应尽量依赖 Python 标准库，避免无必要第三方依赖。
- 保持安全语义清晰：脆弱示例与安全示例必须显式区分，避免误导读者。
- 保持授权边界：内容仅用于教学、防御与授权环境说明，不能把仓库改成攻击利用手册。

## Python 代码风格

### Imports

- 遵循标准库优先、第三方其次、本地模块最后的顺序分组。
- 当前脚本仅使用标准库 `sqlite3`，保持导入简洁，不要为演示代码引入无意义依赖。
- 除非有明显收益，不要使用 `from module import *`。
- 未使用的导入必须删除。

### Formatting

- 遵循 PEP 8 的基本风格，使用 4 空格缩进。
- 一行过长时，优先使用括号换行，而不是反斜杠续行。
- 字符串拼接示例可以保留 f-string 或多段字符串，只要能清楚表现 SQL 结构。
- 代码示例以可读性为先，不要为了“炫技”压缩表达。

### Types

- 当前仓库没有类型检查配置，也没有广泛使用类型注解。
- 新增少量类型注解可以接受，但要保持一致，不要半套 typing 风格混用得很凌乱。
- 如果只改动现有示例，优先沿用当前轻量风格；不要强行把整仓库改造成重类型项目。

### Naming

- 函数名、变量名使用清晰的 `snake_case`。
- 名称要体现教学语义，例如 `vulnerable_login`、`safe_login`、`allowed_fields` 这类命名是推荐方向。
- 避免使用难懂缩写，除非是 SQL、ID 等通用术语。
- 文档文件名保持小写、下划线分隔，延续现有风格，如 `sql_injection_demo.md`。

### Control Flow

- 示例函数应短小、单一职责、便于逐段讲解。
- 演示流程优先显式展开，不要为少量逻辑额外抽象复杂类层次。
- 只有在重复明显、且不会损害可读性时才做抽取复用。

### Error Handling

- 数据库相关操作可以像现有代码一样捕获 `sqlite3.Error`。
- 错误处理的目标是帮助教学观察，而不是隐藏所有问题。
- 面向读者输出时，可返回简短、可理解的错误信息；但不要把真实生产系统的错误暴露做法描述为最佳实践。
- 如果你在文档中谈“推荐实践”，应强调生产环境需要统一异常处理、避免原始数据库报错直出。

### Output And Logging

- 当前示例使用 `print()` 输出 SQL 和结果，这是教学用途的合理做法。
- 不要把 `print()` 教学输出误写成生产日志规范。
- 若新增演示输出，确保标签足够明确，读者能区分 vulnerable/safe 两类路径。

## 文档风格指南

- 仓库以中文内容为主；新增文档默认使用中文，必要术语可保留英文原词。
- 先讲结论，再讲原因，再讲示例，延续现有教学资料的结构化写法。
- 使用明确小节标题和项目符号，方便课堂、讲义和代理读取。
- 涉及风险、攻击、绕过等内容时，必须同时交代防御语境和授权边界。
- 不要引入未经验证的数据、案例或“最新趋势”表述；如需补充事实，请注明来源。

## SQL 与安全相关约定

- 新增 SQL 示例时，必须明确标注哪些是“脆弱写法”，哪些是“安全写法”。
- 值位置优先强调参数化查询。
- 结构位置如排序字段、列名、表名，优先强调白名单映射。
- 不要把危险 payload 作为炫技重点，重点应放在漏洞成因与修复思路。
- 所有安全内容都应服务于防御、审计、教学和授权测试场景。

- 除非任务明确要求，不要重命名大量文件或大规模重排文档结构。

## 与 `.trae` Skill 一致的工作规则

- 处理 PPT、讲义、结构化内容生成任务时，遵循“需求澄清 -> 资料整理/调研 -> 大纲 -> 策划稿 -> 中间产物 -> 复核”的阶段化流程。
- 不要一上来直接生成完整终稿，尤其在需求不清、事实依赖强、面向外部受众时。
- 如果环境无法完成外部调研，应明确说明“结果受限于现有资料”，不要伪造确定性。
- 对高风险任务应设置中间确认点，例如先给研究 brief、提纲或页级策划卡。
- 技术实现方式可灵活，但工作阶段、事实诚实性和可审阅中间产物是硬约束。
